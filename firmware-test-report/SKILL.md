# 固件测试报告生成器

## 报告命名规则

格式:`{设备名称} V{固件版本号}固件第{X}轮测试报告`

- X 用中文数字(一/二/三/四...)
- 示例:`Hub 3 V21.17.17固件第三轮测试报告`

## 报告文件格式

飞书云文档(docx)

---

## 交互模式

### ❗ 第一步:确认报告名称

**无论是创建、更新还是修改测试报告,必须从报告名称开始。**

- 创建新报告:先确认报告名称(设备名称 + 版本号 + 第几轮),再进入引导流程
- 更新已有报告:先确认是哪份报告(通过名称或链接定位),再进行更新
- 报告名称格式:`{设备名称} V{固件版本号}固件第{X}轮测试报告`

### 首次使用
1. 询问基本信息:设备名称、固件版本号、当前第几轮
2. **确认本轮测试起止日期**(用于 bug 轮次分类,即使第一轮也必须确认)
3. 创建飞书云文档,写入**完整框架**(所有章节标题 + 占位 NA)
4. 从第一章第一节开始,**逐步引导**用户填充内容
5. 每完成一节 → 更新文档 → 引导下一节

### 更新已有报告
1. 询问更新哪份报告(获取文档链接)
2. **确认本轮测试起止日期**
3. 读取现有报告内容
4. 检查原文档中与模板无关的内容 → 询问使用者是否保留(说删就全删,严格按模板重写)
5. 根据需要更新对应章节,最终结果必须与模板结构完全一致

### 核心原则:引导填写

**任何章节内容为空或搜索不到时,必须主动引导使用者填写,绝不允许直接写 NA 然后跳过。**

引导规则:
- 每次只引导一个章节,等用户回复后再进入下一个
- 告知用户这个章节需要什么信息,给出具体示例
- 用户明确说"没有"或"NA"时,才写 NA
- 用户说"跳过"时,先写 NA 占位,标记待补充
- **禁止一次性列出多个空字段问用户**,必须一个一个引导,每次只问一个章节
- 能自动获取的(如历史报告、迭代需求文档)先自动搜索,找不到再问用户

### 再次使用(同设备/同项目)
- 已确认过的内容(PRD链接/测试工具/历史报告等)→ **直接复用**,不再询问
- 只引导**新的/变化的**字段(提测单、工时、bug 数据)

## 报告结构

**章节编号规则:**
- 一级标题:一、二、三、四(H1)
- 二级标题:1.1/1.2/2.1/2.2/3.1/3.2 等(H2)
- 所有章节必须带编号,不允许裸标题

### 一、项目介绍(H1)

#### 1.1 参考文档(H2)

有序编号,固定 5 类,没有的标注 NA。
**每个序号单独一行(必须换行)**:

```
1、硬件PRD文档:XXX
2、软件PRD文档:XXX
3、UX设计交互文档:XXX
4、迭代需求文档:XXX
5、其他文档:XXX
```

**填写规则:**
- 硬件PRD → 使用者自己提供链接
- 软件PRD → 使用者自己提供链接
- UX设计交互 → 使用者自己提供链接
- 迭代需求文档 → 可自动搜索飞书文档(按设备名+版本号匹配)
- 其他文档 → 使用者自己提供,没有填 NA

#### 1.2 历史报告(H2)

**获取逻辑(按优先级):**

1. 询问使用者是否有参考测试报告
   - 有 → 读取里面的历史报告链接,复制过来
   - 没有 → 进入 Step 2
2. 自动扫描飞书云文档
   - 使用 `feishu_search_doc_wiki` 搜索
   - 优先匹配新格式:`{设备名称} V{版本号}固件第X轮测试报告`
   - 兼容旧格式:匹配**设备名称** + **固件** + **测试报告**关键词
   - 同一设备的历史版本报告都收集,按版本号排序
   - 列为有序列表,带链接
   - **每个序号单独一行(必须换行)**

#### 1.3 限制条件(H2)

**引导流程:**
1. 询问使用者本轮测试是否遇到环境、条件等限制(如设备不足、网络问题、固件不稳定等)
2. 有 → 填写具体限制
3. 没有 → 写 NA

### 二、测试概要(H1)

#### 2.1 工时投入详情(H2)

**引导流程:**
1. 先查 TOOLS.md "工时记录表（按产品）" 是否已有当前产品的工时表
   - 有 → 直接读取数据，跳到步骤 4
   - 没有 → 进入步骤 2
2. 询问使用者:"是否已经用 work-hours-bitable Skill 生成过工时统计表?"
   - 有 → 使用者给链接 → 读取数据
   - 没有 → 进入步骤 3
3. 建议使用者使用 work-hours-bitable Skill 创建标准工时表
   - 告知优势:"建好后只要按规则填写,以后每次写测试报告我都能自动读取工时,不用再手动填了"
   - 使用者同意 → 触发 work-hours-bitable Skill 创建流程 → 创建完成后让使用者先填工时再回来继续报告
   - 使用者不想创建 → 让使用者自行提供工时信息(手动告知)
4. 读取工时表数据
   - 自动判断表中单位是**小时(h)**还是**人日**
   - 人日换算:总小时数 ÷ 7.5
   - **固件工时拆分规则:**
     - 标签只有"固件" → 当日投入工时全算固件
     - 标签含"固件"+其他(安卓/ios) → 看"固件和APP工时"字段拆分(如"固件4h/安卓2.5h")
     - "固件和APP工时"为空且标签混合 → 无法自动区分,提示使用者补充
   - **工时表格必须包含"测试人"列**(放在固件工时列后面)
     - 从工时表的"测试者"字段读取
     - 同一天有多个人测试 → 写多个人名(如"田东旭、陈娇兰")

**关联 Skill:** `work-hours-bitable`(路径: `~/.openclaw/skills/work-hours-bitable/SKILL.md`）
- 该 Skill 定义了标准 11 字段工时表结构
- 按此结构创建的工时表可被 firmware-test-report 自动读取
- 已有工时表列表见 TOOLS.md "工时记录表（按产品）"

**工时记录表 Bitable 字段结构（标准格式）:**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| 任务名称 | 文本(主字段) | 如“Hub3 V21.17.17 第一轮测试” |
| 日期 | 日期(yyyy/MM/dd) | 当天日期 |
| 当日进展 | 文本 | 当天做了什么 |
| 当日执行用例数 | 数字(0.0) | 执行的用例数量 |
| 标签 | 多选 | 固件/ios/安卓/后台 |
| 固件测试轮次 | 文本 | 如“第一轮”“第二轮”（标签含固件时填写） |
| 当日投入工时 | 数字(0.0) | 小时数 |
| 纯执行用例人力 | 数字(0.0) | 纯执行用例耗时 |
| 固件和APP工时 | 文本 | 标签同时含固件+其他时，分开记录各自工时 |
| 测试者 | 文本 | 测试人员姓名 |
| 产品名称 | 单选 | 如 Hub 3、屋檐灯等 |

**读取工时表规则:**
- 筛选条件：`产品名称` = 当前设备，且 `日期` 在本轮测试起止范围内
- 统计所有测试人员的工时（不只报告作者）
- 按标签分类：固件报告只统计“固件”标签的记录
- 合计小时数后换算人日（÷ 7.5）

**报告中的表格格式(lark-table, 5列):**

| 测试者 | 日期 | 任务内容 | 工时(h) | 人日 |
|--------|------|----------|---------|------|
| xxx | x/x(半天/全天) | xxx第X轮测试 | xx | x.x |
| **合计** | | - | **xx** | **x.x** |

最后一行合计加粗。

#### 2.2 测试范围(H2)

使用者自己填写,默认 NA。

格式:callout 蓝底,简洁列出本轮测试范围。

```html
<callout emoji="clipboard" background-color="light-blue">
**{版本号} 第{X}轮测试范围:**
```

#### 2.3 修改内容详情(H2)

**引导流程:**
1. 请使用者提供**固件转测单截图**(需完整截取转测范围部分)
2. AI 解析截图,**只提取实际修改了什么功能/改了什么东西**(不包含版本号、OTA文件名、文件说明、Gitee链接、Jenkins信息等)
3. 能分模块(BLE/UI/WIFI)就分类统计,转测单未明确分类的就直接复制修改内容

**格式:**

分模块的情况:
```
本次修复/增加内容详情(第{X}轮提测单):
第{X}轮提测,针对第{X-1}轮提测的改动项为

**BLE {芯片型号} 端修改内容:**
1. xxx

**UI {芯片型号} 端修改内容:**
1. xxx

**WIFI {芯片型号} 端修改内容:**
1. xxx
```

未分模块的情况:
```
本次修复/增加内容详情(第{X}轮提测单):

1. xxx
2. xxx
```

没有某个模块则省略该模块。第一轮不写"针对第X-1轮"。

#### 2.4 测试用例(H2)

**引导流程:**
1. 询问使用者是否有测试用例链接(如 ONES 测试用例集 URL)
2. 有 → 填写链接
3. 没有 → 写 NA

#### 2.5 测试计划(H2)

**引导流程:**
1. 询问使用者是否有测试计划链接(如 ONES 测试计划 URL)
2. 有 → 填写链接
3. 没有 → 写 NA

#### 2.6 测试工具(H2)

**引导流程:**
1. 询问使用者本轮使用了哪些测试工具/设备(手机型号及系统版本、拓包工具、第三方平台等)
2. 使用者提供后填入
3. 首次提供后记忆,后续同设备项目可复用(但仍需确认:"上次用的是 xxx,这次一样吗?")

格式:
```
- iOS手机:{型号} ({系统版本})
- Android手机:{型号}
- 第三方平台:Google Home / Apple Home ({设备} {版本}) / Alexa
```

#### 2.7 遗留问题(H2)

默认填:`见项目ONES固件缺陷`

### 三、测试详情(H1)

**第三章需要用户提供 ONES 缺陷数据(Excel 导出)。**

#### 引导流程

1. 询问使用者:"请从 ONES 导出本版本的所有缺陷 Excel"
2. 告知导出操作步骤:
   - 打开 ONES 项目 → 缺陷管理
   - 筛选条件:固件版本 = 当前版本号
   - 点击导出，确保包含以下 10 列:
     **ID / 标题 / 优先级 / 状态 / 固件版本 / 创建者 / 负责人 / 截止日期 / 创建时间 / 更新时间**
   - 导出为 .xlsx 格式
3. 使用者发送 Excel 文件
4. AI 读取解析 → 按轮次分类 → 填充各小节

**引导话术示例:**
> 现在需要你从 ONES 导出本版本的缺陷数据。
> 操作方法: ONES项目 → 缺陷管理 → 筛选「固件版本 = {version}」→ 导出 Excel
> 导出时请确保包含这 10 列: ID、标题、优先级、状态、固件版本、创建者、负责人、截止日期、创建时间、更新时间
> 导出完发给我就行。

#### Excel 输入格式(10列)

| 列 | 字段名 | 说明 |
|----|--------|------|
| A | ID | Bug编号,如 W1091-193520 |
| B | 标题 | Bug描述 |
| C | 优先级 | 普通/重要/紧急 |
| D | 状态 | 关闭/回归通过/待评审/测试提交/持续跟踪 |
| E | 固件版本 | 如 21.17.17 |
| F | 创建者 | 提交人 |
| G | 负责人 | 开发负责人 |
| H | 截止日期 | 可为空 |
| I | 创建时间 | datetime格式 |
| J | 更新时间 | datetime格式 |

#### Bug 轮次判断规则

**根据 bug 创建时间对比各轮次的日期范围来判断。**

轮次日期获取优先级:
1. **工时表** → 任务名称含"第X轮"的记录,对应日期即为该轮次时间范围
2. **工时投入详情**(报告中已填)→ 日期即为本轮测试时间范围
3. 都没有 → 询问使用者本轮测试的起止日期

分类规则:
- bug 创建时间落在本轮日期范围内 → **本轮新增缺陷**
- bug 创建时间在本轮之前 + 状态非"关闭" → **进入回归结果表**

#### 3.1 第{X-1}轮缺陷回归结果(H2)

**第一轮报告无此小节,跳过。**

第二轮及之后显示。

**表格格式(lark-table,8列):**

| ID | 标题 | 上轮状态 | 回归结果 | 负责人 | 创建者 | 创建时间 | 更新时间 |
|----|------|----------|----------|--------|--------|----------|----------|

**回归结果颜色标记:**
- 回归通过 → `<font color="green">回归通过</font>`
- 回归失败 → `<font color="red">回归失败</font>`
- 关闭 → `<font color="green">关闭</font>`
- 测试提交 → `<font color="red">测试提交</font>`
- 持续跟踪 → `<font color="blue">持续跟踪</font>`

表格下方附回归统计:
```
回归统计:共 X 条(回归通过 X / 关闭 X / 测试提交 X / 持续跟踪 X)
```

#### 3.2 新增缺陷(H2)

(第一轮时为 3.1)

**表格格式(lark-table,8列):**

| ID | 标题 | 优先级 | 状态 | 负责人 | 创建者 | 创建时间 | 更新时间 |
|----|------|--------|------|--------|--------|----------|----------|

**状态颜色标记:**
- 关闭 → `<font color="green">关闭</font>`
- 回归通过 → `<font color="green">回归通过</font>`
- 待评审 → `<font color="orange">待评审</font>`
- 测试提交 → `<font color="red">测试提交</font>`
- 持续跟踪 → `<font color="blue">持续跟踪</font>`

表格下方附统计:
```
新增统计:共 X 条(关闭 X / 测试提交 X / ...),优先级分布:紧急 X / 重要 X / 普通 X
```

#### 3.3 缺陷统计(H2)

(第一轮时为 3.2)

显示**全版本**总缺陷数 + 状态分布。

格式:
```
总缺陷数:X 条(第一轮 X + 第二轮新增 X + ...)
```

**饼图:**
- 生成扇形图(饼图),图上标注**百分比和数量**
- 饼图应放在 3.3 缺陷统计标题下方、统计表格上方
- 饼图插入文档后,在饼图下方追加一行提示文字:"【请将上方饼图拖动到 3.x 缺陷统计 标题下方、统计表格上方，然后删除本行提示】"
- 同时在消息中告知使用者拖动位置
- 饼图生成方式:用 HTML/Python matplotlib 生成 PNG,再插入文档

加统计表(饼图下方):

| 状态 | 数量 | 占比 |
|------|------|------|
| 关闭 | X | XX% |
| 回归通过 | X | XX% |
| ... | ... | ... |
| **合计** | **X** | **100%** |

附闭环率:`缺陷闭环率 = (关闭 + 回归通过) / 总数 × 100%`

#### 3.4 风险项(H2)

(第一轮时为 3.3)

列出所有**未闭环** bug(状态非"关闭"且非"回归通过"的)。

**表格格式(lark-table,7列):**

| ID | 标题 | 状态 | 负责人 | 创建者 | 创建时间 | 更新时间 |
|----|------|------|--------|--------|----------|----------|

无未闭环 bug 时写:`无风险项`

#### 3.5 会议纪要(H2)

(第一轮时为 3.4)

默认填 NA。使用者后续补充 bug 评审会议结论。

### 四、测试结论(H1)

根据 bug 状态自动判断:

| 结论 | 条件 | 样式 |
|------|------|------|
| PASS | 所有 bug 关闭/回归通过,或仅剩"持续跟踪" | green callout ✅ |
| 有条件通过 | 少量待评审/测试提交,不阻塞核心功能 | yellow callout ⚠️ |
| 不通过 | 有未解决的阻塞性 bug(测试提交状态) | red callout ❌ |

**格式:**

PASS:
```html
<callout emoji="white_check_mark" background-color="light-green">
**测试通过** 第{X}轮测试完成。总计 X 条缺陷全部关闭/回归通过。
</callout>
```

有条件通过:
```html
<callout emoji="warning" background-color="light-yellow">
**有条件通过** 第{X}轮测试完成。总计 X 条缺陷中...
</callout>
```

不通过:
```html
<callout emoji="x" background-color="light-red">
**测试不通过** 第{X}轮测试完成。总计 X 条缺陷中仍有 X 条未修复...
**遗留未修复(测试提交 X 条):**
- {ID}:{标题}({负责人})
**持续跟踪(X 条):**
- {ID}:{标题}({负责人})
建议针对以上遗留问题进行第{X+1}轮修复。
</callout>
```

## Bug 评审后更新(可选流程)

使用者告知评审结论后:
1. 更新风险项(标注评审结论)
2. 重新评估测试结论(可能从"不通过"升级为"PASS")
3. 填写会议纪要章节

## 记忆机制

Skill 使用过程中,将以下信息存入 memory 供后续复用:
- 设备名称、项目编号
- PRD/UX/需求文档链接
- 测试工具配置
- 历史报告链接列表
